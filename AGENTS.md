# MikanArchive 项目协作说明

## 默认语言

本项目的需求文档、说明文件、注释、界面文案、错误提示、配置说明和后续新增文件默认使用中文。

以下内容可以使用英文：

- 第三方库、API、命令、文件名、协议、代码标识符本身使用英文。
- 面向开源协作、跨系统兼容和工具识别的文件名或目录名。
- 用户明确要求英文输出。

## 项目定位

MikanArchive 是一个个人知识收藏博客，不是普通个人主页。它用于记录教程、笔记、资源收藏、友链、联系入口、个人资料和写作足迹。

核心方向：

- 技术路线：Astro + Markdown/MDX + Tailwind CSS + shadcn 风格组件。
- 视觉气质：以 Firefly / fqzlr.com 公开体验为主要参考的轻二次元收藏馆 + 可读的教程博客 + 简历可展示的专业感。
- 仓库策略：公开框架仓库与私有内容仓库分离。
- 部署策略：第一版优先支持 Cloudflare Pages，Vercel 作为兼容备选。

正式设计依据：

- `docs/superpowers/specs/2026-06-26-mikan-archive-design.md`
- `docs/superpowers/specs/2026-06-27-mikan-archive-firefly-rebuild-design.md`（当前 Firefly 重构方向）
- `docs/superpowers/specs/2026-06-27-mikan-archive-focused-content-layout-design.md`（非主页单列布局与工具导航细化）

## 视觉对比默认规则

- 本项目后续涉及 UI、页面布局、样式、响应式或交互的任务，默认启用浏览器视觉对比，不再重复向用户确认是否启用。
- 视觉对比用于方案确认、实现前后对照和桌面 / 移动端验收；代码与文档修改仍需遵守项目验证、日志和隐私规则。
- `.superpowers/` 仅保存本地视觉讨论草稿，已被 `.gitignore` 忽略，不提交到公开仓库。

## 更新后链接规则

- 每次完成页面、UI、布局、交互、内容展示、导航或可浏览文档相关更新后，最终回复必须提供开发者可以直接打开的对应网页链接。
- 如果本地开发服务正在运行，优先提供 `http://localhost:4321/...` 形式的具体页面链接；不要只给首页，必须列出本次影响页面的具体路径。
- 如果本地开发服务未运行，最终回复必须同时提供启动命令和对应页面路径，例如：

```powershell
npm.cmd run dev -- --host 127.0.0.1 --port 4321
```

- 如果本次变更已经部署到线上，可以同时提供线上链接；如果只是推送到未合并分支，必须明确线上正式站点暂时看不到，需要使用本地链接查看。
- 如果本次更新不影响任何可浏览页面，也要在最终回复中说明“本次没有对应页面链接”。

## 项目专属 Skill

本项目包含以下只服务于 MikanArchive 的项目专属 Skill：

- `.agents/skills/mikan-archive-project/SKILL.md`
- `.agents/skills/mikan-archive-maintenance/SKILL.md`

这些 Skill 只能用于本项目，不能作为全局通用技能使用，也不能服务其他项目。

`mikan-archive-project` 使用场景：

- 更新 MikanArchive 的产品需求、视觉方向、信息架构或内容模型。
- 实现或调整 Astro 页面、内容集合、组件、样式、友链、资源收藏、足迹和内容同步逻辑。
- 维护公开框架仓库与私有内容仓库分离的约定。

`mikan-archive-maintenance` 使用场景：

- 每次修改代码、文档、配置或项目规则后的日志记录、文档同步检查、提交和 push。
- 判断某次更新是普通提交、部署更新，还是需要创建版本 tag。
- 维护 `development-log.md`、`CHANGELOG.md`、README、项目专属 Skill 和部署说明的一致性。

后续如果新增、修改、删除或移动任何项目专属 Skill，必须同步更新本文件中的项目专属 Skill 说明。

## 命名与仓库

- 项目显示名：`MikanArchive`
- 公开框架仓库名：`mikan-archive`
- 私有内容仓库名：`mikan-archive-content`

仓库内说明、配置和示例应使用相对路径，不写死开发者本机绝对路径。本地项目目录以实际克隆或工作区位置为准。

## 第一版范围

顶部导航：

```text
主页 / 文库 / 收藏 / 联系我 / 足迹 / 我的
```

第一版页面：

- 主页：最近文章、精选收藏、足迹小组件、友链预览、简介卡片、可配置背景。
- 文库：文章/教程列表、分类、标签、搜索、归档、草稿过滤。
- 文章详情：MDX 正文、代码高亮、目录、阅读进度、上一篇/下一篇。
- 收藏：一级导航只作为下拉分组，不设收藏总览功能页；下拉直接进入工具导航和摘录收藏，整理分类、标签、收藏理由和适用场景。
- 联系我：集中承载友链、留言和 QQ 群入口；友链保留名片墙、分类筛选和申请说明，公开仓库示例不写入真实私人联系方式。
- 足迹：日历、写作热力图、时间线、最近更新、统计。
- 我的：个人介绍、技能栈、项目入口、简历链接、联系方式。

第一版不做：

- 后台登录管理。
- 网站内一键上传。
- 数据库。
- 用户注册。
- 复杂 CMS。
- 自动友链审核。

## 内容分离规则

公开框架仓库应包含：

- 主题代码。
- 页面布局。
- 组件。
- 样式系统。
- 示例内容。
- 配置模板。
- README、CHANGELOG、文档和脚本。

私有内容仓库应包含：

- 真实文章。
- 图片、背景图、头像和文件资料。
- 资源收藏数据。
- 友链数据。
- 个人资料和简历配置。
- 时间线和更新记录。

公开仓库必须提供 `content.example/`，让没有私有内容仓库的用户也能运行示例站点。

## 文档同步规则

修改任何文件时，都要判断是否影响其他文件或说明。

如果本次修改影响以下内容，必须在同次修改中同步更新相关文件：

- 项目定位、导航、页面范围、内容结构、部署流程或维护流程。
- 入口命令、脚本命令、环境变量、目录结构或模块职责。
- `AGENTS.md` 中记录的项目规则。
- `.agents/skills/*/SKILL.md` 中记录的项目流程。
- README、部署指南、内容仓库说明或实现计划。
- `development-log.md` 或 `CHANGELOG.md` 的记录规则。

检查方向：

- 如果修改了规则文件，检查项目专属 Skill 是否需要同步。
- 如果修改了项目专属 Skill，检查 `AGENTS.md` 是否需要同步。
- 如果修改了内容模型，检查示例内容、校验脚本、README 和设计/计划文档是否需要同步。
- 如果修改了部署流程，检查 `.env.example`、README、维护 Skill 和实现计划是否需要同步。
- 如果修改了导航或页面功能，检查设计文档、README、组件和测试是否需要同步。

## 下一步任务交接

仓库使用 `docs/next-tasks.md` 记录跨会话的待执行任务、实施步骤、阻塞点和下一步动作。

它与 `development-log.md` 的区别：

- `development-log.md` 记录已经完成的事实、涉及文件和验证结果。
- `docs/next-tasks.md` 记录接下来要做什么、做到哪一步、下一次从哪里继续。

规则：

- 每次开始修改代码、文档、配置、规则或项目专属 Skill 前，必须先读取 `docs/next-tasks.md`。
- 每次新增目标、拆分步骤、暂停任务、遇到阻塞或跨会话交接时，必须更新 `docs/next-tasks.md`。
- 已完成并验证的步骤可以从 `docs/next-tasks.md` 删除，或临时标记为已完成；同一目标全部完成、验证、记录、提交和 push 后，应清空对应任务内容，只保留文档使用规则。
- `docs/next-tasks.md` 不替代 `development-log.md`、CHANGELOG 或设计文档；它只服务于当前和下一次继续执行。
- 不得把私有内容、密钥、本机 token、cookie 或真实私人资料写入 `docs/next-tasks.md`。

## 开发记录

后续每次生成或修改代码、文档、配置、规则或项目专属 Skill，都需要同步更新 `development-log.md`。

记录内容应包括：

- 修改日期和具体时间。
- 修改范围。
- 涉及文件。
- 具体内容。
- 验证情况。
- 如果没有运行验证或测试，需要说明原因。

日常开发日志标题只使用日期时间，不写版本号。

只有在明确创建版本 tag、正式发布一个站点版本或用户要求记录版本时，才使用：

```markdown
## Version x.x.x - YYYY-MM-DD HH:mm:ss +08:00
```

MikanArchive 是静态网站项目，不需要 LaunchDock 那种桌面安装包 Release 流程。普通代码或文档更新不自动创建 GitHub Release。

## 更新与提交规则

普通更新流程：

1. 读取 `AGENTS.md` 和相关项目专属 Skill。
2. 修改代码、文档、配置或规则。
3. 同步更新 `development-log.md`。
4. 根据影响面检查并同步 README、`AGENTS.md`、项目专属 Skill、设计文档、实现计划、部署说明和示例内容。
5. 运行与改动风险匹配的验证。
6. 检查 `git status --short` 和关键 diff。
7. 提交并 push 到 GitHub。

只有用户明确要求创建 tag、正式发布版本、生成 Release 或发布说明时，才执行版本发布流程。意图不明确时必须先向用户确认。

## 质量要求

- 优先保护真实内容仓库，不把私有文章、私有图片、token、cookie、账号密码或部署密钥写入公开仓库、日志或 README。
- 如果引入或改造 CuteLeaf/Firefly 主题代码，必须保留 Firefly 与 fuwari 的 MIT 版权声明和清晰致谢。
- 可以复刻 fqzlr.com 的公开布局体验和交互结构，但不能复制该博主的真实文章、头像、图片、个人资料、私有配置或账号文案。
- JSON/YAML/Markdown 等结构化内容使用结构化格式维护，不用临时字符串拼接伪造结构。
- 示例内容不能冒充真实内容。
- 文章页阅读体验优先，装饰动效不能影响教程阅读。
- UI 不能全站只使用单一粉色，应有粉、蓝、暖黄、薄荷绿和中性色的平衡。
- 顶部导航保持简洁，不把所有入口都平铺出来。
- 文件名和目录名使用简洁英文；面向用户的文档内容默认中文。
- 每次完成工作前说明运行过的验证；没有验证时说明原因。
