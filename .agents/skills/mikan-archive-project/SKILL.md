# MikanArchive 项目技能

## 使用原则

- 只服务当前 MikanArchive 项目，不迁移为全局通用技能。
- 默认使用中文编写文档、说明、注释、界面文案和错误提示。
- 新增文件和目录使用简洁英文名称；文件内容仍默认中文。
- 开源文档、配置和示例使用仓库相对路径，不写死开发者本机绝对路径。
- 处理项目需求或代码前，先读取根目录 `AGENTS.md` 和正式设计文档。
- 当前视觉与主题重构方向以 `docs/superpowers/specs/2026-06-27-mikan-archive-firefly-rebuild-design.md` 为基础，非主页布局和工具导航以 `docs/superpowers/specs/2026-06-27-mikan-archive-focused-content-layout-design.md` 为最新细化依据；旧设计文档保留为项目初始背景。
- 后续涉及 UI、页面布局、样式、响应式或交互的任务，默认启用浏览器视觉对比，不再重复询问用户是否启用；视觉草稿保存在已忽略的 `.superpowers/` 目录。
- 每次完成页面、UI、布局、交互、内容展示、导航或可浏览文档相关更新后，最终回复必须提供开发者可以直接打开的对应网页链接；本地服务运行时优先给 `http://localhost:4321/...` 的具体页面路径，未运行时同时给启动命令和页面路径。
- 如果相关更新只推送到未合并分支，最终回复必须说明线上正式站点暂时看不到，需要通过本地链接查看；如果本次更新不影响可浏览页面，也要明确说明没有对应页面链接。
- 每次开始修改代码、文档、配置或规则前，先读取 `docs/next-tasks.md`，确认当前待办、阻塞点和下一步动作。
- 每次新增目标、拆分步骤、暂停任务或跨会话交接时，更新 `docs/next-tasks.md`；已完成并验证的步骤可以删除或临时标记为已完成。
- 每次生成或修改代码、文档、配置或规则时，同步更新根目录 `development-log.md`。
- 每次修改后检查 README、`AGENTS.md`、项目专属 Skill、设计文档、实现计划、部署说明和示例内容是否仍与当前代码结构、入口命令、内容模型、环境变量和功能行为一致；如不一致或可能误导用户，必须同次更新。
- 维护流程、提交、push、版本 tag 或部署相关事项使用 `.agents/skills/mikan-archive-maintenance/SKILL.md`。
- 每次新增、修改、删除或移动项目专属 Skill 时，同步更新根目录 `AGENTS.md`。
- 不把私有内容仓库中的真实文章、图片、token、cookie、账号密码或部署密钥写入公开仓库。

## 项目目标

MikanArchive 是一个个人知识收藏博客。它面向教程记录、资源收藏、友链、个人资料和写作足迹，不做成普通个人主页。

第一版目标：

- 使用 Astro 静态生成。
- 使用 Markdown/MDX 管理文章。
- 以 CuteLeaf/Firefly 的 Astro 静态博客主题为底座或强参考重建博客壳层、壁纸、导航、侧栏、文章列表和文章页体验。
- 可引入 Firefly 依赖的 Svelte islands、Pagefind、壁纸模式、侧栏组件和显示设置；保留 MikanArchive 的内容分离层。
- 支持公开框架仓库和私有内容仓库分离。
- 优先支持 Cloudflare Pages 部署。

## 信息架构

顶部导航固定为：

```text
主页 / 文库 / 收藏 / 友邻 / 足迹 / 我的
```

页面职责：

- 主页：整体入口，展示最近文章、精选收藏、足迹小组件、友邻预览、简介卡片和可配置背景。
- 文库：文章/教程、分类、标签、搜索、归档和草稿过滤。
- 文章详情：MDX 正文、代码高亮、目录、阅读进度、上一篇/下一篇。
- 收藏：通过收藏总览分流到工具导航与摘录收藏；工具按分类整理长期入口，摘录记录来源、适用场景、收藏日期、标签和外部链接。
- 友邻：友链名片墙、分类筛选、申请指南弹窗和复制本站信息。
- 足迹：日历、写作热力图、时间线、最近更新和统计。
- 我的：个人介绍、技能栈、项目入口、简历链接和联系方式。

第一版不实现后台登录、网站内一键上传、数据库、用户注册、复杂 CMS 和自动友链审核。

## 内容分离

公开框架仓库：

```text
主题代码
页面布局
组件
样式系统
示例内容
配置模板
README / CHANGELOG / docs / scripts
```

私有内容仓库：

```text
真实文章
图片、背景图、头像和文件资料
资源收藏数据
友链数据
个人资料和简历配置
时间线和更新记录
```

私有内容仓库推荐结构：

```text
content/
  posts/
  resources/resources.json
  links/friends.json
  links/sites.json
  profile/about.md
  profile/resume.json
  records/timeline.json
  records/updates.json
  assets/images/
  assets/backgrounds/
  assets/avatars/
  assets/files/
```

公开仓库必须保留 `content.example/`，用于本地开发、开源展示和没有私有内容同步时的兜底构建。

## 视觉与交互约束

视觉关键词：

```text
轻二次元收藏馆 + 知识书架 + 可读教程博客
```

要求：

- 以 Firefly / fqzlr.com 的公开博客体验作为当前重构目标，复刻布局结构和交互体验，不复制私人内容和素材。
- 如果使用 Firefly 代码或组件设计，保留 Firefly 与 fuwari 的 MIT 版权声明和页脚/文档致谢。
- 使用柔和粉、浅蓝、暖黄、薄荷绿和中性色，避免全站单一粉色。
- 首页、收藏和友邻页可以更有个性；文章详情页必须更克制。
- 支持可配置背景图、首页横幅、玻璃导航、徽章、状态点、头像卡片、资源卡片、日历和热力图。
- 交互可包含页面淡入、卡片微浮起、标签 hover、搜索弹窗、友链申请弹窗、回到顶部和阅读进度。
- 装饰动画不得影响文章阅读、移动端排版和按钮/文字可读性。
- 顶部导航保持简洁，不把所有入口平铺出来。

## 修改文件时的检查清单

处理本项目需求或代码时，至少检查：

- 是否仍符合正式设计文档。
- 是否保持公开框架仓库与私有内容仓库分离。
- 是否避免泄露真实私有内容或密钥。
- 是否需要更新 `content.example/`。
- 是否需要更新内容校验规则或同步脚本。
- 是否需要更新 README、部署指南或 `.env.example`。
- 是否需要更新 `docs/next-tasks.md`，记录下一步、已完成步骤或阻塞点。
- 是否需要更新 `AGENTS.md` 或项目专属 Skill。
- 是否需要更新 `development-log.md`。
- 如果涉及页面或组件，是否考虑桌面和移动端。
- 如果涉及文章页，是否优先保证阅读体验。
